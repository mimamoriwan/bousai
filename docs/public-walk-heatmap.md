# 公開お散歩ヒートマップのプライバシー設計

## 目的

本人専用の正確なお散歩記録を公開せず、複数の利用者が散歩している地域の傾向だけを「にぎわい」として地図に表示します。

## 公開条件

- 正確な座標を約250m四方のセル中心へ丸める。
- 週単位で集約する。
- 同じセル・同じ週に3人以上の利用者が記録した場合だけ公開する。
- 同じ利用者が何度記録しても、公開条件の人数は1人として数える。
- 公開する項目はセル中心、合計件数、参加人数、週ID、更新時刻だけとする。

## 公開しない情報

- Firebase UIDやユーザーハッシュ
- 元の正確な緯度・経度
- 個別の記録時刻
- 行動の順番や移動ルート
- 3人未満のセル

## データの流れ

1. クライアントは本人専用の`walkActions`へ正確な記録を保存する。
2. Cloud Functionsが作成・削除イベントを受け取り、週・約250mセルへ集約する。
3. UID別件数と3人未満の集約は、クライアントから読めない内部コレクションへ保存する。
4. 3人以上になったセルだけを`walkHeatmapPeriods/{week}/cells`へ複製する。
5. 公開マップは当週と前週の公開セルだけをヒートレイヤーとして表示する。

## 安全策

- 公開・内部集約コレクションへのクライアント書き込みは禁止する。
- 記録の`createdAt`はFirestoreの受付時刻と一致させ、任意の週への記録偽装を防ぐ。
- FunctionsイベントはイベントIDで重複処理を防ぐ。
- 記録削除時は集約件数も減らし、3人未満になったセルは公開コレクションから削除する。
- Functionsの純粋ロジック、Firestore Security Rules、実際のFirestoreトランザクションをCIで検査する。

## デプロイについて

この機能はCloud FunctionsとFirestore Security Rulesの同時反映が必要です。開発PRをマージしても自動では本番へデプロイしません。本番用Rulesとの整合、古い週の内部集約・イベント記録を定期削除する方針、対象と反映順を実機確認後に決めてから明示的にデプロイします。
